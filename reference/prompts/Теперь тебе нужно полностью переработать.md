## Задача

Разработать модульную JavaScript/TypeScript-библиотеку для управляемой стилизации текста с поддержкой нескольких языков, детерминированной случайностью и подключаемыми фильтрами.

Библиотека должна принимать текст, параметры преобразования и набор активных фильтров, после чего возвращать преобразованный текст. Преобразование должно быть воспроизводимым: одинаковый вход, seed, язык и настройки всегда дают одинаковый результат. При плавном изменении интенсивности преобразования результат должен меняться постепенно, без хаотичных “скачков”.

Библиотека должна быть построена по принципу KISS: простая основа, понятные интерфейсы, минимум скрытой магии. Расширяемость достигается не сложной архитектурой, а стабильными контрактами для языков, фильтров, словарей и текстовых операций.

---

## Основные цели

Библиотека должна поддерживать минимальную обработку любого языка там, где это возможно: определение языка, Unicode-сегментация, разбиение на токены, безопасная работа с пробелами, пунктуацией, emoji и разными письменностями.

Для отдельных языков должна быть расширенная поддержка через языковые пакеты. На первом этапе необходимо реализовать английский и русский языки. Каждый язык должен подключать свои словари, правила, NLP-адаптеры и фильтры.

Архитектура должна позволять добавлять новый язык без изменения ядра библиотеки. Новый язык должен подключаться как отдельный модуль с конфигурацией и наборами данных.

Все преобразования должны быть реализованы как независимые фильтры. Фильтр — это отдельный модуль, который получает текстовую структуру, контекст выполнения и свои настройки, после чего возвращает изменённую структуру. Фильтры можно включать, отключать, переупорядочивать и конфигурировать отдельно.

## Библиотеки, которые нужно использовать через интерфейсы

Библиотека не должна напрямую размазывать вызовы сторонних пакетов по коду. Для них нужны адаптеры.

`franc` используется для определения языка текста. Он возвращает ISO 639-3 код языка и может вернуть `und`, если язык не определён. Также важно учитывать ограничение: на коротких текстах `franc` может ошибаться, поэтому язык должен можно быть передан вручную пользователем. ([GitHub][1])

`intl-segmenter` используется как обёртка над `Intl.Segmenter` для разбиения текста на графемы, слова или предложения. Это базовый слой для Unicode-безопасной сегментации, особенно для длинных строк и не-ASCII текста. ([GitHub][2])

`compromise` используется для английского NLP: части речи, match-паттерны, числа, предложения, существительные, глаголы и другие операции, которые нужны фильтрам английского языка. ([GitHub][3])

`ru-compromise` используется как языковой NLP-слой для русского языка. Он должен быть подключён через тот же интерфейс NLP-адаптера, что и английский `compromise`, но без требования, чтобы оба адаптера имели абсолютно одинаковые возможности. ([GitHub][4])

## Ключевой принцип

Ядро библиотеки не должно знать, как именно устроены английский, русский или любой другой язык.

Ядро должно знать только это:

```ts
interface LanguagePack {
  code: string
  aliases: string[]
  segmenter?: SegmenterAdapter
  nlp?: NlpAdapter
  dictionaries: LanguageDictionaries
  filters: Filter[]
  rules: LanguageRules
}
```

А конкретный язык уже решает, какие словари, правила и NLP-возможности у него есть.

## Общий pipeline

Рекомендуемый порядок работы:

```text
input text
→ normalize input
→ detect or resolve language
→ segment text
→ build document model
→ run filters in configured order
→ render document back to text
→ output text
```

Ядро не должно работать с текстом как с простой строкой после начальной стадии. Лучше использовать промежуточную структуру документа.

Минимальная структура:

```ts
interface TextDocument {
  original: string
  language: string
  tokens: Token[]
  metadata: Record<string, unknown>
}
```

Токен:

```ts
interface Token {
  id: string
  text: string
  normalized: string
  kind: 'word' | 'space' | 'punctuation' | 'number' | 'emoji' | 'other'
  start: number
  end: number
  tags?: string[]
  meta?: Record<string, unknown>
}
```

Важно сохранять пробелы и пунктуацию как токены. Это упростит обратную сборку текста и снизит риск поломки форматирования.

## Фильтры

Фильтр — основная единица преобразования.

```ts
interface Filter {
  id: string
  phase: 'pre' | 'lexical' | 'syntactic' | 'post'
  languages?: string[] | 'any'
  apply(document: TextDocument, context: TransformContext): TextDocument
}
```

Контекст:

```ts
interface TransformContext {
  seed: string
  intensity: number
  language: string
  rng: DeterministicRandom
  options: Record<string, unknown>
  resources: LanguageResources
}
```

Примеры фильтров:

```text
dictionaryReplacementFilter
misspellingFilter
phraseInsertionFilter
numberSimplificationFilter
casePreservationFilter
punctuationFilter
grammarMistakeFilter
transliterationFilter
emojiFilter
customRegexFilter
```

Фильтр должен быть независимым. Если фильтр отключён, остальные не должны ломаться. Если для языка нет нужного NLP-адаптера, фильтр должен либо пропускаться, либо использовать fallback-режим.

## Детерминированная случайность

Обычный `Math.random()` использовать нельзя.

Нужен общий deterministic RNG, зависящий от:

```text
global seed
filter id
token id
rule id
```

Пример идеи:

```ts
rng.float(`${seed}:${filterId}:${token.id}:${ruleId}`)
```

Это нужно, чтобы решение для конкретного токена было стабильным. Тогда при изменении интенсивности с `0.41` до `0.42` не будет полного пересчёта случайности.

Правило применения должно быть таким:

```ts
const roll = rng.float(key) // число от 0 до 1
const shouldApply = roll < probability * intensity
```

Тогда при росте `intensity` изменения только добавляются. Например, если у токена `roll = 0.37`, то он не изменится при `intensity = 0.3`, но изменится при `intensity = 0.4`. Уже изменённые токены не будут хаотично возвращаться назад, если интенсивность плавно растёт.

Для выбора варианта из массива тоже нужен стабильный выбор. Но его лучше отделить от решения “применять или нет”:

```ts
const shouldApplyRoll = rng.float(`${baseKey}:apply`)
const variantRoll = rng.float(`${baseKey}:variant`)
```

Так изменение интенсивности не должно менять выбранный вариант у уже активированной замены.

---

## Управление интенсивностью

Интенсивность должна быть числом, обычно от `0` до `1`, но архитектура может разрешать значения выше `1`.

Каждый фильтр должен иметь собственный scale:

```ts
interface FilterConfig {
  enabled: boolean
  intensity?: number
  probability?: number
  options?: Record<string, unknown>
}
```

Итоговая вероятность:

```text
globalIntensity × filterIntensity × ruleProbability
```

Это позволит отдельно усилить, например, словарные замены, но ослабить вставки фраз.

## Формат словарей

Данные должны быть разделены по языкам и типам. Не нужно держать всё в одном огромном JSON.

Рекомендуемая структура:

```text
data/
  en/
    replacements.json
    misspellings.json
    insertions.json
    numbers.json
    morphology.json
    regex-rules.json
  ru/
    replacements.json
    misspellings.json
    insertions.json
    numbers.json
    morphology.json
    regex-rules.json
```

Каждый JSON должен быть пригоден для пакетной обработки: простой формат, стабильные поля, без JS-логики внутри.

Пример `replacements.json`:

```json
{
  "version": 1,
  "language": "en",
  "entries": [
    {
      "source": "okay",
      "variants": [
        { "value": "ok", "weight": 2 },
        { "value": "oki", "weight": 1 }
      ],
      "tags": ["informal"],
      "probability": 0.5
    }
  ]
}
```

Пример `misspellings.json`:

```json
{
  "version": 1,
  "language": "en",
  "entries": [
    {
      "source": "because",
      "variants": [
        { "value": "becuz", "weight": 2 },
        { "value": "bc", "weight": 1 }
      ],
      "probability": 0.4
    }
  ]
}
```

Пример `regex-rules.json`:

```json
{
  "version": 1,
  "language": "en",
  "rules": [
    {
      "id": "final-s-to-z",
      "pattern": "s\\b",
      "replacement": "z",
      "probability": 0.25
    }
  ]
}
```

Для русского языка правила будут другими:

```json
{
  "version": 1,
  "language": "ru",
  "rules": [
    {
      "id": "что-to-што",
      "pattern": "\\bчто\\b",
      "replacement": "што",
      "probability": 0.4
    }
  ]
}
```

## Разделение универсального и языкового поведения

Универсальные фильтры могут работать для любого языка:

```text
unicode segmentation
case preservation
whitespace preservation
punctuation manipulation
emoji insertion
basic dictionary replacement
basic regex replacement
```

Языковые фильтры работают только при наличии нужных данных или NLP:

```text
English POS-based insertion
Russian POS-based insertion
English number verbalization
Russian morphology-aware replacement
language-specific misspellings
language-specific grammar mistakes
```

Это важное правило: “любой язык” означает не одинаковое качество для всех языков, а безопасный fallback.

## Языковые пакеты

Каждый язык должен экспортировать пакет:

```ts
export const englishPack: LanguagePack = {
  code: 'en',
  aliases: ['eng'],
  segmenter: createIntlSegmenterAdapter('en'),
  nlp: createCompromiseAdapter(),
  dictionaries: loadEnglishDictionaries(),
  filters: [
    dictionaryReplacementFilter,
    englishPhraseInsertionFilter,
    englishNumberFilter
  ],
  rules: englishRules
}
```

```ts
export const russianPack: LanguagePack = {
  code: 'ru',
  aliases: ['rus'],
  segmenter: createIntlSegmenterAdapter('ru'),
  nlp: createRuCompromiseAdapter(),
  dictionaries: loadRussianDictionaries(),
  filters: [
    dictionaryReplacementFilter,
    russianPhraseInsertionFilter,
    russianMisspellingFilter
  ],
  rules: russianRules
}
```

Новый язык должен добавляться так:

```ts
engine.registerLanguage(myLanguagePack)
```

Без изменения ядра.

## Публичный API

Минимальный API:

```ts
const engine = createTextTransformEngine({
  seed: 'user-seed',
  languages: [englishPack, russianPack],
  defaultLanguage: 'en'
})

const result = engine.transform(text, {
  language: 'auto',
  intensity: 0.65,
  filters: {
    dictionaryReplacement: { enabled: true, intensity: 1 },
    misspelling: { enabled: true, intensity: 0.8 },
    phraseInsertion: { enabled: false }
  }
})
```

Результат лучше возвращать не только строкой:

```ts
interface TransformResult {
  text: string
  language: string
  appliedFilters: string[]
  changes?: TextChange[]
  warnings?: string[]
}
```

`changes` полезны для отладки, тестов и UI.

```ts
interface TextChange {
  filterId: string
  tokenId?: string
  from: string
  to: string
  reason?: string
}
```

## Требования к модульности

Ядро библиотеки не должно импортировать конкретные словари английского или русского языка.

Фильтр не должен читать файлы напрямую. Он должен получать данные через `context.resources`.

NLP-адаптеры должны быть заменяемыми. Если завтра вместо `compromise` появится другой NLP-инструмент, фильтры не должны переписываться полностью.

Словари должны быть данными, а не кодом.

Порядок фильтров должен быть явно задан в конфигурации.

Каждый фильтр должен иметь unit-тесты отдельно от всего pipeline.

## MVP

Для первой версии не нужно делать всё сразу.

Минимальный MVP:

```text
TypeScript core
deterministic RNG
language detection through franc
segmentation through intl-segmenter
language registry
filter pipeline
English language pack
Russian language pack
dictionary replacement filter
regex misspelling filter
basic insertion filter
configurable intensity
debug changes output
```

Для английского можно подключить `compromise` и сделать POS-aware вставки. Для русского сначала достаточно словарей, regex-правил и базовой сегментации; `ru-compromise` подключить через адаптер и использовать только там, где его возможностей достаточно.

## Что не входит в первую версию

Не нужно сразу делать полноценную морфологию для всех языков.

Не нужно пытаться одинаково качественно поддержать все языки.

Не нужно строить сложную plugin-систему с dependency injection framework.

Не нужно делать ML.

Не нужно делать один универсальный словарь для всех языков.

Не нужно смешивать правила, словари и код фильтров в одном файле.

## Критерии готовности

Библиотека считается готовой на уровне архитектурного MVP, если:

Один и тот же текст с одним seed всегда даёт один и тот же результат.

При увеличении intensity изменения добавляются постепенно, а не перескакивают хаотично.

Английский и русский подключены как отдельные language packs.

Словари английского и русского физически разделены.

Фильтры можно включать и отключать независимо.

Отсутствие NLP для языка не ломает универсальные фильтры.

Можно добавить новый язык без изменения ядра.

Можно добавить новый фильтр без изменения существующих фильтров.

Можно получить список изменений для отладки.
